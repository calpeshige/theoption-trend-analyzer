# コミュニティ AI学習データ共有システム — 実装計画（最終版）

## 概要

拡張機能ユーザーが「自動バックアップ機能」を有効化することで、自身のAI学習データが定期的にローカルへバックアップされると同時に、コミュニティへも提供される仕組み。提供されたデータは集約・整形され、`/manual` ページから他ユーザーがダウンロードして取り込めるようになる。

## 設計の核心コンセプト

「**自動バックアップ機能**」をフロントに立てることで、ユーザーが自分自身のメリット（データ消失からの保護）と引き換えに、自然にコミュニティへも貢献する仕組み。

```
ユーザー視点: 「自動バックアップ」=自分のためにやる行為
                    ↓
       同時に「コミュニティにも貢献」される（オプトイン同意）
                    ↓
        提供する動機が「自分のメリット」と一致
```

## 全体アーキテクチャ

```
[拡張機能ユーザーA]
  ├ 初回起動時にウェルカムダイアログで「自動バックアップ」を選択
  ├ 7日に1回（推奨）、ランダム時刻にポップアップで実行確認
  ├ 許可されると:
  │   ・JSONエクスポート → ユーザーのDLフォルダに保存（ローカルバックアップ）
  │   ・同データを Apps Script Webhook へチャンク分割送信（コミュニティ提供）
  └ 取引中はポップアップを出さず、待機時間まで延期
                ↓
[Google Apps Script Webhook]
  ├ チャンク受信 & サイズ検証
  └ 送信元のuploadIdごとに Drive へ保存
                ↓
[Google Drive 5TB（あなたのアカウント）]
  ├ TheOption-Uploads フォルダに一時保管
  └ チャンクファイルが集まる
                ↓
（毎日 03:00 JST 自動実行）
                ↓
[GitHub Actions ワークフロー]
  1. Drive APIでフォルダ内ファイル一覧取得
  2. 同一uploadIdのチャンクを結合 → 1つのgzipファイルに復元
  3. 各ファイルをDL & gunzip & 構造検証
  4. 通貨ペア別にMap統合（timestamp重複排除）
  5. 新しい順 20,000件にトリミング
  6. メタデータ抽出（期間、件数、時間帯分布）
  7. gzip圧縮して GitHub Releases にアップロード
  8. trading-manual.html のマーカー間を自動更新
  9. Drive上の処理済みファイルを削除（容量解放）
  10. 自動コミット & プッシュ
                ↓
[Vercel 自動再デプロイ]
                ↓
[拡張機能ユーザーB] ◀── /manualページから.json.gzを手動DL ── [/manual ダウンロードタブ]
                  ↓
          拡張機能のJSONインポート（gzip対応版、時間帯フィルタ付き）
                  ↓
              IndexedDBに統合
```

## 仕様確定事項

### バックアップ頻度
- **デフォルト**: 7日に1回（週1回）
- **ユーザー選択肢**: 3日 / 7日 / 14日
- 設定パネルから変更可能

### 提供方式
- **オプトイン**: 初回起動時のウェルカムダイアログで明示的同意
- **送信タイミング**: バックアップ実行時に同時送信
- **送信内容**: フルデータ（差分ではない）
  - 理由: ユーザーのローカルバックアップ価値を保つため
- **バックアップ単位**: 通貨ペアごと
  - ローカルDL: 全通貨ペアのデータが個別ファイルでDLフォルダへ
  - コミュニティ提供: 同データを送信

### 集中回避（同日集中問題対策）
- **初回オフセット**: 有効化から 3〜10日の範囲でランダム分散
- **時刻ジッター**: 設定時刻から ±2時間のランダム
- **起動時遅延**: 0〜10分のランダム遅延
- **uploadId**: ユーザーごとにランダム生成、同日集中を完全分散

### 許可制ポップアップ
- **取引中は出さない**: 判定時間中・カウントダウン10秒以内・シグナル発生直後はポップアップ抑止
- **延期可能**: 最大7日まで延期可能
- **永久拒否可能**: 30日経過後はリマインダー停止
- **「後で」の限度**: 5回連続「後で」を選ぶと最終確認、それでも拒否ならOFFのまま継続使用可能

### データ品質チェック
システム自身がエクスポートしたデータ前提のため、**構造検証のみ**実施:
- キーが `theoption_ml_*` プレフィックス
- 値が配列形式
- 各レコードに `timestamp` / `price` フィールド必須
- timestamp が過去5年〜現在の範囲内

### サイズ・上限
- 1ユーザーあたり想定データ量: 約500MB（圧縮後、25,000件規模）
- Apps Script の1リクエスト上限: 約50MB → **30MBチャンク分割**で対応
- 1ファイル上限: 2GB（GitHub Releases制約と一致）
- 統合後のトリミング: **新しい順 20,000件**（通貨ペアごと）

### 配信
- **GitHub Releases**（既存システムと同じ仕組み）
- 通貨ペア単位でファイル分割
- リポジトリ: `calpeshige/theoption-releases`
- タグ命名規則: `data-{ASSET}-latest`（例: `data-USDJPY-latest`）

### ダウンロード時のフィルタリング
- 取得時にユーザーが時間帯を選択可能（メタデータの hourCoverage を活用）
- 全時間帯/取引時間帯のみ/カスタム選択

### 更新頻度
- GitHub Actions cron: **毎日 03:00 JST**（UTC 18:00）
- 手動実行も可能（`workflow_dispatch`）

### スパム対策
- 自動バックアップなので人為的スパム不可
- ランキング機能は実装しない（個人内表示のみで承認欲求充足）

3. 以下のコードを貼り付け（FOLDER_ID を更新）:

```javascript
const FOLDER_ID = 'XXXXX'; // 受信箱フォルダID
const MAX_CHUNK_SIZE = 40 * 1024 * 1024; // 40MB（安全マージン込み）

function doPost(e) {
  try {
    const params = JSON.parse(e.postData.contents);

    // チャンクモード: チャンク単位で受信
    const filename = `${params.uploadId}_chunk_${String(params.chunkIndex).padStart(4, '0')}_of_${String(params.totalChunks).padStart(4, '0')}_${params.assetName.replace('/', '')}.bin`;

    // サイズチェック
    const decodedSize = (params.dataBase64.length * 3) / 4;
    if (decodedSize > MAX_CHUNK_SIZE) {
      return jsonResponse({ success: false, error: 'Chunk too large' });
    }

    const folder = DriveApp.getFolderById(FOLDER_ID);
    const blob = Utilities.newBlob(
      Utilities.base64Decode(params.dataBase64),
      'application/octet-stream',
      filename
    );

    folder.createFile(blob);
    return jsonResponse({ success: true, filename });
  } catch (err) {
    return jsonResponse({ success: false, error: err.toString() });
  }
}

function jsonResponse(obj) {
  return ContentService
    .createTextOutput(JSON.stringify(obj))
    .setMimeType(ContentService.MimeType.JSON);
}
```

#### 2-4. バックアップ実行関数（[theoption-analyzer.js](../theoption-analyzer.js) に追加）

```javascript
const COMMUNITY_WEBHOOK_URL = 'https://script.google.com/macros/s/.../exec';
const CHUNK_SIZE = 30 * 1024 * 1024; // 30MB

async function performAutoBackup() {
  const dbManager = new DBManager();
  await dbManager.init();

  const assetList = await dbManager.getAssetList();
  if (assetList.length === 0) return;

  // ユニークID生成（同一バックアップセッションを識別）
  const uploadId = `${Date.now()}_${Math.random().toString(36).substr(2, 9)}`;

  for (const { assetName } of assetList) {
    // 1. ストリーミングエクスポート
    const result = await dbManager.streamExport(assetName);
    if (!result) continue;

    // 2. ローカル保存（ユーザーのDLフォルダへ）
    await downloadToUser(result.blob, `theoption_backup_${assetName.replace('/','_')}_${new Date().toISOString().slice(0,10)}.json`);

    // 3. gzip圧縮
    const compressed = await compressBlob(result.blob);

    // 4. チャンク分割してWebhook送信
    await uploadInChunks(uploadId, assetName, compressed);
  }

  // 5. 最終バックアップ日時を記録
  chrome.storage.local.set({ lastBackupTime: Date.now() });

  // 6. 次回予定を計算（ランダムジッター付き）
  scheduleNextBackup();
}

async function uploadInChunks(uploadId, assetName, compressedBlob) {
  const totalSize = compressedBlob.size;
  const totalChunks = Math.ceil(totalSize / CHUNK_SIZE);

  for (let i = 0; i < totalChunks; i++) {
    const start = i * CHUNK_SIZE;
    const end = Math.min(start + CHUNK_SIZE, totalSize);
    const chunkBlob = compressedBlob.slice(start, end);
    const base64 = await blobToBase64(chunkBlob);

    await fetch(COMMUNITY_WEBHOOK_URL, {
      method: 'POST',
      body: JSON.stringify({
        uploadId,
        chunkIndex: i,
        totalChunks,
        assetName,
        dataBase64: base64
      })
    });
  }
}
```

```yaml
name: Curate Community Data

on:
  schedule:
    - cron: '0 18 * * *'  # 毎日 03:00 JST
  workflow_dispatch:

jobs:
  curate:
    runs-on: ubuntu-latest

      contents: write
    steps:

      - uses: actions/setup-node@v4

      - run: npm install googleapis

      - name: Run curation script
        env:
          GOOGLE_SERVICE_ACCOUNT_JSON: ${{ secrets.GOOGLE_SERVICE_ACCOUNT_JSON }}
          DRIVE_FOLDER_ID: ${{ secrets.DRIVE_FOLDER_ID }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: node scripts/curate-from-drive.js

      - name: Commit changes
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add docs/trading-manual.html data-meta/
          git diff --staged --quiet || git commit -m "🤖 Auto-update community data $(date -u +%Y-%m-%d)"
```

## ユーザー体験フロー

### 初回（インストール直後）

```
1. ユーザーが拡張機能をインストール
2. 拡張機能起動
3. ウェルカムダイアログ表示:
   「自動バックアップ機能を有効化しますか？
    7日に1回、PCにバックアップが保存され、
    コミュニティにも共有されます」
   [✓ 有効化（推奨）] [今は有効にしない]
4. 「有効化」を選んだ場合:
   - 初回バックアップ予定: 3〜10日後のランダム日時
5. 「今は有効にしない」を選んだ場合:
   - 通常通り使用可能
   - 5日後・30日後にリマインダー
   - 30日経過後はリマインダー停止
```

### 定期実行時

```
1. 起動時に予定時刻チェック
2. 予定時刻を過ぎていて、取引中でなければ:
   ポップアップ表示:
   「バックアップを実行しますか？
    USD/JPY: 25,234件
    EUR/USD: 12,500件」
   [今すぐ実行] [1日後に延期] [3日後に延期]
3. 「今すぐ実行」:
   バックグラウンドで実行
   - ローカルJSONを自動DL
   - コミュニティ提供（チャンク分割送信）
4. 完了通知（控えめ）:
   「✅ バックアップが完了しました」
```

### 取引中の場合

```
1. 取引判定時間中・カウントダウン10秒以内・シグナル発生直後:
   ポップアップは出さない
2. 待機時間に入ったら、改めてポップアップ表示
3. ユーザーは取引を妨げられない
```

## セキュリティ留意事項

| 項目 | 対策 |
|---|---|
| Webhook URL流出 | 公開前提。Apps Script側でサイズ・構造検証 |
| 巨大ファイル攻撃 | チャンク40MB上限 + 1ファイル2GB上限 |
| 改ざんデータ混入 | timestamp範囲チェック（過去5年〜現在）|
| 提供同意 | 初回ウェルカムで明示的同意 + 設定でいつでも変更可 |
| サービスアカウント漏洩 | GitHub Secretsで暗号化保管、最小権限（Drive編集のみ）|
| 失敗通知 | GitHub Actions失敗時にメール通知（GitHub標準機能）|
| プライバシー | 個人特定情報は送信しない（データ内容のみ）|

## 議論の経緯（参考メモ）

設計に至るまでに検討した重要論点:

1. **「データ提供ボタン」を押すユーザーがいない問題** → 自動バックアップとセットにすることで動機を一致させた
2. **特定時間帯ユーザーへの害** → 提供と取得を分離、取得時に時間帯フィルタを提供
3. **データ取得側のメリット問題** → そもそも提供できる人は取得不要なので、貢献動機を「承認・利他性」に集約
4. **ランキングによるスパム** → ランキング廃止、個人内表示のみ
5. **アップロード時の重さ** → チャンク分割 + ストリーミング処理 + 取引中回避
6. **完全自動化のプライバシー問題** → オプトイン同意 + 許可制ポップアップ
7. **GitHub Releases 直接アップロード不可** → 認証トークン埋め込み問題、Drive中継が安全
8. **同日集中問題** → ランダムオフセット + ジッター + uploadId 分散
9. **頻度の最適解** → 7日（週1回）がバランス最適、3日/14日も選択可
10. **容量問題** → Drive 5TB + 即削除運用で 50,000ユーザーまでスケール可
